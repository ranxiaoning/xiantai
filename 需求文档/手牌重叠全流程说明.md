# 手牌重叠及变灰判断全流程说明

本文档将详细向您解释游戏中与“卡牌变灰”以及“超量卡牌重叠”有关的整个代码流程中的每一个环节和细节，确保您可以清晰地读懂游戏逻辑是如何运转的。

## 一、 卡牌变灰（不可用）判断流程

您在测试时发现有些卡牌呈现灰色、无法打出，这是因为游戏战斗引擎引擎对资源和可用性进行了一次校验。

1. **引擎判断逻辑发生在哪里？**
   在 `scripts/BattleEngine.gd` 中，有一个方法叫 `can_play_card(card: Dictionary) -> bool:`。
   ```gdscript
   func can_play_card(card: Dictionary) -> bool:
       if s["phase"] != "player":
           return false # 不是玩家的回合，绝对不可打出
       # 比较当前玩家灵力是否大于等于卡牌需要的灵力，以及道慧是否足够
       return s["player_ling_li"] >= card.get("ling_li", 0) and \
              s["player_dao_hui"] >= card.get("dao_hui", 0)
   ```
   **细节：** 这里就是判断根源，如果不满足条件，函数会返回 `false`。

2. **视图如何接收状态变灰指令？**
   在 `scripts/BattleScene.gd` 生成手牌时调用了以下逻辑：
   ```gdscript
   func _make_card_view(card: Dictionary) -> Control:
       var view: Control = CardViewScene.instantiate()
       var can_play: bool = _engine.can_play_card(card) 
       # 此时从上方函数计算出了 can_play 的真假(true/false)
       ...
       view.setup(card, tex, not can_play) # not can_play 代表 disabled = true
       ...
   ```

3. **真实的视觉变灰**
   在单卡视图 `scripts/CardView.gd` 中，执行 setup：
   ```gdscript
   func setup(data: Dictionary, texture: Texture2D, disabled: bool) -> void:
       ...
       $Dimmer.visible = disabled # 激活灰色遮罩节点
   ```
   **结论：** 在之前的版本，卡牌变灰是通过 `$Dimmer.visible = true` 渲染一个灰色遮层。经过您的要求改进后，我们移除了这个碍眼且不够明亮的视觉反馈。但内部控制拦截依然存在，当资源不足时仅仅是不响应您的点击或拖拽，让界面一直保持漂亮。

## 二、 超过五张卡重叠且附带丝滑效果的流程

在废弃掉原先的 Godot 自带 `HBoxContainer` 后，我们将布局节点改为无主导布局规则的自由节点 `Control`，从而让卡牌不仅能在数量多时互相重叠展现扇形，而且能拥有“移动平滑、旋转圆润”的极致体验机制。

1. **生成与复用触发点**
   在 `scripts/BattleScene.gd` 中每次玩家行动触发界面刷新，会进入 `_update_hand_display()`。

2. **保留老卡并飞出弃牌**
   我们不再一刀切地 `queue_free()` 把手牌杀掉然后生硬地重新创建。我们会检测如果某张原来在桌面上的卡，在新一回合数据中没有了（被打出或者丢弃了），我们就为其实例化一个飞升渐隐且消逝的 `Tween` 退场动画。

3. **计算动态坐标（核心环节）**
   对于还需要显示的卡牌（含抓上来的新卡）：
   ```gdscript
   var count = new_children.size()
   var sep = 8.0
   if count > 5:
       # 如果总数超过5张，我们需要挤压它们：
       var visual_max = 532.0 
       sep = (visual_max - count * card_w) / (count - 1.0)
       
   var total_w = count * card_w + (count - 1) * sep
   var start_x = (max_w - total_w) / 2.0
   
   for i in range(count):
       # 我们按数组坐标排布，保持水平平行（Y坐标恒定为40）以及角度0.0
       var t_pos = Vector2(start_x + i * (card_w + sep), 40)
       # 告知卡牌平滑飞过新位置
       view.move_to(t_pos, 0.0) 
   ```

4. **鼠标放入弹出查看（悬停展示前置）**
   当多张卡牌丝滑入场并折叠压贴在一起后，在 `scripts/CardView.gd` 中有交互的悬停重置：
   ```gdscript
   func _on_entered() -> void: 
       z_index = 10 
       # 当悬停时：我们将卡单提至最高层“10”，无视身边的遮挡
       var to_pos = anim_pos + Vector2(0, -20)
       _animate(HOVER_SCALE, to_pos, 0.0) # 让卡不但放大1.1倍，而且平滑地悬空垂直升起 20px 以供查看全图。
   
   func _on_exited() -> void: 
       z_index = 0
       # 降维，恢复到常规列表里的原重叠层级身份，平滑缩放并归位。
       _animate(NORMAL_SCALE, anim_pos, anim_rot)
   ```

## 总结
通过以上三步联动：**逻辑资源卡扣** -> **数量判定重叠** -> **Z层级弹射查看**，我们就完成了一个高效、具有极佳打牌体验的基础交互体系，同时也配备了完善的单元测试保证数值不越界。
